# カレンダーアプリ

## アプリ概要

このアプリは日付の表示や予定の追加/編集/削除などの一般的なカレンダーアプリが持つ機能およびUIの実装の練習を目的に作られました｡

使用した技術については主にReactとFirebaseを使用しています｡

このアプリでは認証機能などの実装は行っていませんので､以下のurlからすぐ触って確かめることが可能です｡
<https://sakanobu-calendar.firebaseapp.com>

## アプリケーションの機能一覧

- カレンダー表示
- 予定の追加(タイトル､メモ､時間､タグ)
- 予定の編集および削除
- タグによる一部の予定の表示および非表示の取捨選択
- 他の年月のカレンダーからの本日のカレンダーへと一気にジャンプ可能
- 指定した年月のカレンダーへと一気にジャンプ可能

## カレンダーアプリ作成の動機

インターネットで話題になるIT業界未経験者の就職活動成功体験記を調べていたところ､自分には作りたいアプリケーションのアイディアや現実世界の身の回りの課題などでピンと来るものがありませんでした｡

そこで､以下の2つの理由からフロントエンドの技術を中心にしてカレンダーアプリを作ることに決めました｡

1. 形あるモノの作成に長けているフロントエンドという領域の技術を磨きたい
1. フロントエンドの練習としてカレンダーアプリは丁度いいレベル感である

### 1について

プログラミングを学んでいて常々感じていたのは､｢人は文字や図がないと全く思考が進まない｣ということでした｡

というのも､ひとえに機能を実装するといってもそこには要件定義に基づいた仕様の策定/DBとそのテーブルの設計/UIの設計/アプリ全体の設計/コーディング…etcと多岐に渡って考慮しなければならず､そのような中で頭だけで考えようとしてしまうと曖昧な部分への懸念や場合分けで頭がいっぱいになって堂々巡りになってしまう…ということを何度も経験したからです｡

加えて､実際の業務であればその問題は個人に留まらず､チームのメンバーやITに疎いかもしれない顧客などの個々の人間の間で認識がズレているせいで手戻りや諍いが生まれるといったことも想像せざるを得ません｡

そのような問題意識から人々の認識のすり合わせに向いているITの領域は何かと考えたとき､やはりUIという視覚的に明らかなものの作成を生業とするフロントエンドが魅力的に映り､TypeScriptやReactといった技術を使ったアプリ開発を行おうと思い立ちました｡

### 2について

1で述べた理由からTypeScriptやReactの勉強を進めてToDoリストを作成する中で以下の記事を見かけました｡

[2020年にWebフロントエンドを勉強する人が作るべきたったひとつのアプリ - Qiita](https://qiita.com/Dragon-taro/items/03f322dee15b19c33613)

この記事の本文にあるようにカレンダーはほど良いコンポーネントやStateの量を持ち､かつ､ToDoリストと異なり日付系のライブラリにも着手しなければならないので丁度いい題材だなと思いました｡

冒頭で述べたとおり私には作り出したいオリジナルアプリは特に無く､また､フロントエンドの練習になるような経験を積みたいという気持ちが強かったので､ムリにアイディアをひねり出そうとはせずカレンダーアプリを実装することにしました｡

本来の業務では推奨されないとは思いますが経験を積みたい技術ありきで開発を進めていったため､TypeScriptを用いたReactの開発//Custom Hooksの練習/(ほとんどの機能は使えてませんが)Firebaseを用いたBaaSによる開発/ESLint, Prettier, Jestなどの環境構築…etcなどと試行錯誤できたのは良かったと思います｡

## 各種技術の選定理由

### なぜTypeScriptを使ったのか?

カレンダーアプリ作成の動機という項目で｢人は文字や図がないと全く思考が進まない｣という私の関心について述べましたが､思考云々以前にそもそも人間はミスをしやすい生き物です｡

タイプミス､引数や戻り値のミスなどは当然頻発し､エラーの文章をもとに原因となる箇所を探すのも骨が折れます｡

そして､こういった小さなトラブルを放置してしまえば時間や集中力が削られていき余計にミスを生むなど負のサイクルに陥ってしまうのは想像に難くなく､これではいくら思考を前に進める施策を打ったとしてもその効果は薄いでしょう｡

これらのことから､私の関心は｢思考を促す環境づくり｣だけでなく｢集中を乱すものを出来るだけ削る｣ということも対象になり､ITエンジニアなのだから機械に任せられることは出来るだけ任せてしまおうとまずはプログラミング言語そのものに静的な型チェックの仕組みを持つTypeScriptを選択しました｡

### なぜReactを使ったのか?

素のJavaScriptやTypeScriptでフロントエンドを実装していくのは辛いのため､QiitaやZennなどで今も盛んに記事が投稿されているReactとVue.jsのどちらか一方を選ぼうと当初から半ば決めつけていました｡
両者の公式サイトや紹介記事を読む中で､ちょうど技術書典9で全3巻に及ぶReactの技術同人誌である『[りあクト！ TypeScriptで始めるつらくないReact開発](https://techbookfest.org/product/5436045076201472?productVariantID=6240785991204864)』が出版されると聞いて飛びついてしまい､そのままの流れでアプリ開発もReactにしてしまいました｡
今振り返ってみるととても安直な決め方であったなと思いますが､とりあえず学んで触ってみないことには比較もできず足踏みするばかりなので結果的には仕方なかったかなとも考えています｡
